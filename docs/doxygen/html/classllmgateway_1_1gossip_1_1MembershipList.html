Detailed Description

Thread-safe membership list with SWIM incarnation-based conflict resolution.

Incarnation resolution rules:

  • Higher incarnation always wins for the same member.
  • At equal incarnation: DEAD > SUSPECT > ALIVE (stronger state wins).
  • A member can refute a SUSPECT about itself by incrementing its incarnation.

◆ ApplyUpdate()

bool llmgateway::gossip::MembershipList::ApplyUpdate ( const MembershipUpdate & update)

Apply a membership update.

Returns true if the update changed state. This is the core conflict resolution logic.

-— Incarnation-based conflict resolution -— Rule 1: Higher incarnation always wins (including higher DEAD). Rule 2: At equal incarnation, stronger state wins (DEAD > SUSPECT > ALIVE). Rule 3: A strictly older incarnation is always stale and ignored, even if it claims DEAD. This is what lets a refuted node stay ALIVE: after self-refute bumps its incarnation, stale DEAD updates from piggyback buffers of uninformed peers still fail the incarnation check and don't resurrect.

Stale update — ignore regardless of state. This includes stale DEAD, which would otherwise re-kill an already-refuted node.

Same incarnation: stronger state wins. State ordering: ALIVE(0) < SUSPECT(1) < DEAD(2)

Same or weaker state — update load info only.

Fire the callback on a same-state load change so that downstream subscribers (e.g., the gateway's ReplicaRegistry) can refresh capacity / model-version info. Without this, subscribers would cache whatever metadata arrived with the first state-transition update forever.

Stronger state at same incarnation — fall through to apply.

Higher incarnation or stronger state — fall through to apply.

Apply the update.

Queue for piggybacking.

Fire callback if state actually changed.

Release lock before callback to avoid deadlocks.
